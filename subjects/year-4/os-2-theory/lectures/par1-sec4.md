### 1. مسألة المنتج والمستهلك (Producer-Consumer Problem) — الحل الكامل بالمسيّرات

**النص الأصلي يقول:** تُدعى هذه المسألة أيضاً بمسألة المخزن المحدود (`Bounded-Buffer Problem`). يمكن حل هذه المسألة باستخدام المسيّرات كما يلي: نستخدم المسيّرات التالية: المسيّر الثنائي `mutex` والذي يأخذ القيمة الابتدائية 1، المسيّر `full` يأخذ القيمة الابتدائية 0، المسيّر `empty` ويأخذ القيمة الابتدائية N (سعة المخزن).

**الشرح المبسّط:**
هذا هو الحل **الرسمي والكامل** لمشكلة `Lost Wakeup` التي ظهرت سابقاً مع `Sleep/Wakeup` الساذج. بدلاً من متغير `count` واحد يُفحص يدوياً، نستخدم **ثلاثة مسيّرات** لكل منها دور محدد تماماً:
- `empty` (Counting Semaphore = N): يعدّ عدد الأماكن الفارغة المتبقية في المخزن.
- `full` (Counting Semaphore = 0): يعدّ عدد العناصر الممتلئة الجاهزة للاستهلاك.
- `mutex` (Binary Semaphore = 1): يحمي عملية الإدراج/السحب الفعلية من التداخل في حال وجود أكثر من منتج أو مستهلك واحد.

**💡 التشبيه:**
> تخيل مستودعاً بسعة N صندوق: `empty` هو عداد "الصناديق الفارغة المتاحة"، و`full` هو عداد "الصناديق الممتلئة الجاهزة للنقل"، و`mutex` هو مفتاح الباب الذي يسمح لعامل واحد فقط بالدخول للمستودع في نفس اللحظة لترتيب الصناديق.
> **وجه الشبه:** الصناديق الفارغة = `empty`، الصناديق الممتلئة = `full`، مفتاح الباب = `mutex`.

#### مهم للامتحان ⚠️:
> يجب دائماً أن يُستدعى `wait(empty)` أو `wait(full)` **قبل** `wait(mutex)` وليس العكس، وإلا قد يحدث استعصاء (سيُشرح لاحقاً بالتفصيل ضمن "مشاكل المسيّرات").

### 2. عملية المنتج (Producer Process)

**النص الأصلي يقول:**
```
do {
    // produce an item in next_produced
    wait (empty);   /* decrement empty count */
    wait (mutex);   /* enter critical region */
    // add the item to the buffer
    signal (mutex); /* leave critical region */
    signal (full);  /* increment count of full slots */
} while (TRUE);
```

**الشرح المبسّط:**
المنتج ينتظر أولاً وجود مكان فارغ (`wait(empty)`)، فإن نجح ذلك يدخل المنطقة الحرجة (`wait(mutex)`) ليضيف العنصر بأمان (بدون تداخل مع منتج آخر)، ثم يخرج من المنطقة الحرجة (`signal(mutex)`) ويُخبر المستهلكين بوجود عنصر جديد (`signal(full)`).

```c
// Producer process
do {
    produce_item();      // produce an item in next_produced
    wait(empty);          // decrement empty count; block if buffer full
    wait(mutex);           // enter critical region
    insert_item();         // add the item to the buffer
    signal(mutex);          // leave critical region
    signal(full);            // increment count of full slots
} while (TRUE);
```

#### شرح كل سطر:
1. `wait(empty);` → يتحقق من وجود مكان فارغ؛ إن لم يوجد (`empty==0`) يُحجب المنتج تلقائياً وبدون استهلاك معالج.
2. `wait(mutex);` → يدخل المنطقة الحرجة المتعلقة بهيكل المخزن نفسه (لحمايته من منتج/مستهلك آخر يعمل بالتوازي).
3. `insert_item();` → إضافة العنصر فعلياً داخل المخزن.
4. `signal(mutex);` → تحرير القفل عن المخزن فوراً بعد الانتهاء من الإدراج (وليس بعد إرسال `full`، لتقليل زمن الحجز).
5. `signal(full);` → إخبار أي مستهلك محجوب بأن هناك عنصراً جديداً متاحاً.

**الناتج المتوقع:**
> عدد العناصر في المخزن لا يتجاوز أبداً N، ولا يحدث أي تداخل بين عمليتي إدراج متزامنتين، ولا تُفقد أي إشارة إيقاظ بفضل ذرّية `wait`/`signal` داخل بنية المسيّر.

### 3. عملية المستهلك (Consumer Process)

**النص الأصلي يقول:**
```
do {
    wait (full);    /* decrement full count */
    wait (mutex);   /* enter critical region */
    // remove an item from buffer to next_consumed
    signal (mutex); /* leave critical region */
    signal (empty); /* increment count of empty slots */
    // consume the item in next_consumed
} while (TRUE);
```

**الشرح المبسّط:**
بالتماثل التام مع المنتج: المستهلك ينتظر أولاً وجود عنصر جاهز (`wait(full)`)، ثم يدخل المنطقة الحرجة لسحب العنصر بأمان، ثم يخرج ويُخبر المنتجين بأن مكاناً جديداً أصبح فارغاً (`signal(empty)`)، وأخيراً يستهلك العنصر خارج المنطقة الحرجة (لأن الاستهلاك الفعلي لا يحتاج حماية المخزن المشترك).

```c
// Consumer process
do {
    wait(full);            // decrement full count; block if buffer empty
    wait(mutex);            // enter critical region
    item = remove_item();    // remove an item from buffer to next_consumed
    signal(mutex);            // leave critical region
    signal(empty);             // increment count of empty slots
    consume_item(item);         // consume the item in next_consumed
} while (TRUE);
```

#### شرح كل سطر:
1. `wait(full);` → يتحقق من وجود عنصر جاهز؛ إن لم يوجد يُحجب المستهلك تلقائياً.
2. `wait(mutex);` → دخول المنطقة الحرجة لحماية عملية السحب.
3. `item = remove_item();` → سحب العنصر فعلياً من المخزن.
4. `signal(mutex);` → تحرير القفل فوراً.
5. `signal(empty);` → إخبار أي منتج محجوب بوجود مكان فارغ جديد.
6. `consume_item(item);` → استهلاك العنصر (خارج المنطقة الحرجة عمداً لتقليل زمن حجز `mutex`).

#### الفهم الخاطئ الشائع ❌: يمكن استدعاء `consume_item(item)` قبل `signal(mutex)` طالما أن العنصر سُحب أصلاً.
#### الفهم الصحيح ✅: تُستهلك العملية عادة **بعد** تحرير `mutex` تحديداً لتقليل زمن حجز المخزن المشترك عن باقي العمليات المنتظرة، رغم أن ترتيبها لا يُغيّر الصحة المنطقية بل الأداء فقط.

#### 🤔 تفعيل الفهم (اسأل نفسك):
> **سؤال:** لماذا نحتاج `mutex` أصلاً في هذا الحل، رغم وجود `empty` و`full`؟
> **لماذا هذا مهم؟** لأن `empty` و`full` يضبطان **العدد الإجمالي** للعناصر/الأماكن فقط، لكن لا يمنعان منتجين اثنين (أو مستهلكين اثنين) من تعديل بنية المخزن الداخلية (كمؤشر الإدراج) في نفس اللحظة؛ لذلك `mutex` ضروري لحماية العملية الفيزيائية للإدراج/السحب نفسها.

### 4. مسألة القرّاء والكتّاب (Readers and Writer Problem)

**النص الأصلي يقول:** لتكن لدينا بيانات مشتركة بين مجموعة عمليات بحيث نميز نوعين بين هذه العمليات: القرّاء: يقومون بقراءة هذه البيانات دون إجراء أي تعديلات على هذه البيانات. الكتّاب: يمكن أن يقوموا بقراءة وكتابة البيانات المشتركة. نود السماح لعدة قرّاء بأن يقوموا بالقراءة في نفس الوقت بينما يسمح لكاتب واحد فقط بالتعامل مع البيانات المشتركة دون السماح لأي عملية أخرى بالوصول إلى هذه البيانات، أي بحيث أن عمليات القراءة لا تقوم بالانتظار إلا إذا كان هناك عملية كتابة ما تقوم بتعديل على البيانات المشتركة. المطلوب استعمال المسيرات في كتابة نص برمجي يسمح بذلك لكل نوع من العمليات (القراء + الكتاب).

**الشرح المبسّط:**
هذه مسألة تزامن مختلفة جوهرياً عن `Producer-Consumer`: هنا **نوعان من العمليات بحقوق وصول مختلفة**:
- عدة قرّاء يمكنهم العمل **بالتوازي التام** لأن القراءة لا تُغيّر البيانات، فلا خطر من تداخلهم.
- كاتب واحد فقط يجب أن يُمنع أي عملية أخرى (قارئ أو كاتب) من الوصول للبيانات أثناء كتابته.

القاعدة الذهبية: **القرّاء لا ينتظرون إلا بسبب كاتب فعلي يعمل حالياً**، وليس بسبب قرّاء آخرين.

**💡 التشبيه:**
> تخيل مكتبة فيها كتاب واحد مشترك: يمكن لعدة أشخاص قراءته في نفس الوقت (تصوير أو نسخ صفحات) دون مشكلة، لكن إن أراد شخص تعديل محتوى الكتاب فعلياً (الكتابة عليه)، يجب أن يُخرج الجميع ويعمل بمفرده.
> **وجه الشبه:** القرّاء المتزامنون = عمليات القراءة المتوازية، الكاتب المنفرد = عملية الكتابة الحصرية.

#### 📊 المخطط: نمط وصول القرّاء والكاتب للبيانات المشتركة

#### ما هذا المخطط؟
> يوضّح أن قاعدة البيانات المشتركة يمكن أن تُستخدم إما من قبل عدة قرّاء معاً، أو من قبل كاتب واحد فقط حصرياً، مع منع أي قارئ من الدخول أثناء وجود كاتب.

#### وصف العُقد:
| # | العُقدة | النوع `kind` | الشرح |
| --- | --- | --- | --- |
| 1 | `Shared Data` | مورد مشترك | قاعدة البيانات أو المتغير المشترك |
| 2 | `Reader (R)` | عملية قراءة | يمكن لعدة قرّاء الوصول معاً |
| 3 | `Writer (W)` | عملية كتابة | كاتب واحد فقط يُسمح له بالوصول الحصري |

#### وصف الروابط:
| من | إلى | التسمية | نوع السهم | الشرح |
| --- | --- | --- | --- | --- |
| عدة `Reader` | `Shared Data` | وصول متزامن | سماح | يمكن لعدة قرّاء الدخول معاً بلا قيود متبادلة |
| `Writer` واحد | `Shared Data` | وصول حصري | منع | لا يُسمح لأي عملية أخرى بالدخول أثناء الكتابة |

### 5. حل مسألة القرّاء والكتّاب باستخدام المسيّرات

**النص الأصلي يقول:** لحل المسألة نستخدم البينات المشتركة التالية: مسيّر ثنائي `wrt` يأخذ القيمة الابتدائية 1. عدد صحيح `readcount` يأخذ القيمة الابتدائية 0. مسيّر ثنائي `mutex` يأخذ القيمة الابتدائية 1. فتكون عملية الكاتب (Writer Process):
```
do {
    wait (wrt);
    // writing is performed
    signal (wrt);
} while (TRUE);
```
وتكون عملية القارئ (Reader Process):
```
do {
    wait (mutex);      /* Allow 1 reader in entry */
    readcount ++;
    if (readcount == 1)
        wait (wrt);   /* 1st reader locks writer */
    signal (mutex)
        // reading is performed
    wait (mutex);
    readcount --;
    if (readcount == 0)
        signal (wrt); /* last reader frees writer */
    signal (mutex);
} while (TRUE);
```

**الشرح المبسّط:**
هذا الحل يستخدم ثلاث بنى بيانات:
- `wrt`: مسيّر يمثل "قفل الكتابة"، الكاتب يحجزه بالكامل طوال فترة كتابته.
- `readcount`: عدّاد عدد القرّاء النشطين حالياً.
- `mutex`: يحمي التعديل على `readcount` نفسه من التداخل بين قرّاء متعددين.

**الفكرة الذكية:** بدلاً من أن يحجز كل قارئ `wrt` بمفرده (مما يمنع التوازي بين القرّاء)، فإن **القارئ الأول فقط** (`readcount==1`) هو من يحجز `wrt` نيابة عن كل القرّاء، و**القارئ الأخير فقط** (`readcount==0` بعد الإنقاص) هو من يحرره. القرّاء بينهما يدخلون بحرية دون التعامل مع `wrt` إطلاقاً.

```algorithm
1 | دخول قارئ | wait(mutex) ثم readcount++ | يزيد عداد القرّاء النشطين بأمان
2 | حجز الكاتب عند أول قارئ | if(readcount==1) wait(wrt) | القارئ الأول يمنع أي كاتب من الدخول
3 | تحرير mutex | signal(mutex) | يسمح لقرّاء آخرين بالدخول بالتوازي فوراً
4 | القراءة الفعلية | reading is performed | خارج حماية mutex، بالتوازي مع قرّاء آخرين
5 | خروج قارئ | wait(mutex) ثم readcount-- | ينقص عداد القرّاء بأمان
6 | تحرير الكاتب عند آخر قارئ | if(readcount==0) signal(wrt) | آخر قارئ يسمح لكاتب منتظر بالدخول أخيراً
7 | تحرير mutex | signal(mutex) | إنهاء العملية بأمان
```

#### نقاط التنفيذ:
- إذا استمر وصول قرّاء جدد بلا انقطاع، فقد ينتظر الكاتب لفترة طويلة جداً أو إلى الأبد — وهذا مثال كلاسيكي على مشكلة **المجاعة (`Starvation`)** التي تتعرض لها عملية الكاتب في هذا الحل تحديداً (يُسمى هذا التحيز لصالح القرّاء `readers-priority`).

#### مهم للامتحان ⚠️:
> هذا الحل يُعطي **أفضلية للقرّاء** على حساب الكاتب؛ في حال الحاجة لحل أكثر عدلاً (`fair` أو حتى أفضلية للكاتب)، يجب تعديل الخوارزمية بإضافة مسيّر ثالث يمنع القرّاء الجدد من الدخول إن كان هناك كاتب ينتظر بالفعل.

### 6. استعمال المسيّرات مع عدة موارد متماثلة (Counting Semaphore عام)

**النص الأصلي يقول:** يمكن استعمال المسيّرات للتحكم في استعمال عدة موارد `K` مشتركة تتسابق عليها عدة عمليات `N`. نعرّف مسيّراً عداداً ونعطيه القيمة الابتدائية `k` (عدد الموارد المتوفرة). هذا من شأنه أن يسمح لـ `k` عملية بالدخول إلى مناطقها الحرجة بآن واحد، كل منها يستعمل أحد الـ `k` مورد المتوفرة. عندما تريد أي عملية استعمال أحد الموارد فإنها تنتظر دورها في الحصول عليه باستدعاء `wait()` مخفضة بذلك عداد المسيّر. عندما تحرر أي عملية مورداً انتهت من استعماله فإنها تستدعي `signal()` لزيادة قيمة عداد المسيّر. عندما يكون العداد صفرياً (قيمته صفر) فهذا يعني أن جميع الوارد مستعملة، وأن أية عملية تود الحصول على مورد فعلياً لا تنتظر حتى يتحرر أحد الموارد لى الأقل (يصبح العداد أكبر من الصفر).

**الشرح المبسّط:**
هذا هو الاستخدام **العام والأشمل** لـ `Counting Semaphore`: بدل التركيز على مشكلة إقصاء تبادلي واحدة، هنا نتحكم في **مجموعة من K مورداً متطابقاً** (مثل K اتصال قاعدة بيانات، أو K طابعة، أو K مقعد في صالة انتظار). القيمة الابتدائية = عدد الموارد الكلي، وكل `wait` يعني "خذ مورداً واحداً"، وكل `signal` يعني "أعد مورداً واحداً".

**💡 التشبيه:**
> يشبه هذا نظام تأجير دراجات في مدينة بعدد K دراجة: كل شخص يستأجر دراجة (`wait`) ينقص العداد بواحد، وكل من يعيدها (`signal`) يزيد العداد بواحد. إن وصل العداد للصفر، يجب الانتظار حتى يُعيد أحدهم دراجة.
> **وجه الشبه:** عدد الدراجات المتاحة = قيمة المسيّر.

### 7. الاستعصاء، المجاعة، وعكس الأولوية (Deadlock, Starvation, Priority Inversion)

**النص الأصلي يقول:** الاستعصاء (`Deadlock`): نقول عن عمليتين أو أكثر أنها في حالة استعصاء إذا كانت كل عملية في حالة انتظار لحدث ما ولا يمكن لهذا الحدث أن يتحقق إلا من قبل عملية أخرى من نفس مجموعة العمليات التي في حالة انتظار. مثال:
```
P0                          P1
wait(S);                   wait(Q);
wait(Q);                   wait(S);
.                           .
.                           .
signal(S); signal(Q);      signal(Q); signal(S);
```
المجاعة (`Starvation`): هي الدخول في حالة انتظار لا نهائي، بحيث لا يتم نقل العملية من رتل الانتظار للمسيّر إلى حالة الجاهزية. عكس الأولوية (`Priority Inversion`): هي مشكلة جدولة فيها عملية ذات أولوية أدنى قفلاً أو مسيّراً تكون عملية ذات أولوية أعلى بحاجة للحصول عليه.

**الشرح المبسّط:**
- **الاستعصاء (`Deadlock`):** ينتج هنا تحديداً من **عكس ترتيب حجز المسيّرات** بين عمليتين. `P0` تحجز `S` ثم تحاول حجز `Q`، بينما `P1` تحجز `Q` ثم تحاول حجز `S`. إن نُفذت `wait(S)` من `P0` و`wait(Q)` من `P1` في نفس الوقت تقريباً، ستنتظر كل عملية المورد الذي حجزته الأخرى — **انتظار دائري (`circular wait`)** لا ينفك أبداً.
- **المجاعة (`Starvation`):** على عكس الاستعصاء، هنا العملية **قد تُوقظ نظرياً** لكن لا يحدث ذلك فعلياً أبداً بسبب سياسة اختيار غير عادلة (كأن تُفضّل قائمة الانتظار عمليات أخرى بشكل مستمر).
- **عكس الأولوية (`Priority Inversion`):** مشكلة جدولة وليست مشكلة استعصاء أو مجاعة مباشرة؛ عملية منخفضة الأولوية تحجز مورداً (مسيّراً) تحتاجه عملية عالية الأولوية، فتُجبر العملية عالية الأولوية على الانتظار — وهو أمر منطقي حتى الآن، لكن المشكلة تتفاقم إن تدخلت عملية **متوسطة** الأولوية وسحبت المعالج من العملية منخفضة الأولوية، فتطول مدة انتظار العملية عالية الأولوية بشكل غير متناسب مع أولويتها.

**💡 التشبيه (Deadlock):**
> يشبه هذا شخصين يعبران جسراً ضيقاً من اتجاهين متعاكسين، كل منهما يمسك بحبل من طرف ويحتاج الطرف الآخر (الذي يمسكه الآخر) ليكمل عبوره — فلا يتحرك أي منهما إلى الأبد.
> **وجه الشبه:** المسيّر `S` والمسيّر `Q` = طرفا الحبل الممسوكين بالتبادل.

#### مهم للامتحان ⚠️:
> شرط الاستعصاء هنا هو **ترتيب معكوس** لحجز الموارد بين العمليتين (`P0`: S ثم Q — `P1`: Q ثم S). القاعدة الذهبية لتجنبه: **يجب أن تحجز جميع العمليات الموارد المشتركة بنفس الترتيب دائماً** (Resource Ordering).

### 8. مشكلة عكس الأولوية بالتفصيل (Priority Inversion) وسقف الأولوية (Priority Ceiling)

**النص الأصلي يقول:** على فرض أن مسلكاً ذا أولوية منخفضة قام بقفل الـ `Mutex` مانعاً مسلكاً ذا أولوية أعلى من الحصول عليه (بالتالي لا يمكن للمسلك ذي الأولوية الأعلى أن يستخدمه حتى يقوم المسلك الآخر بفك القفل عنه)، فإذا أتى مسلك ذو أولوية متوسطة (بين المسلكين) وهو يريد أن ينفذ على منطقة ليست ضمن المنطقة الحرجة (وبالتالي يطلب جدولة من قبل المعالج ليدخل ضمن التنفيذ) فإذا كان هذا المسلك يحتوي على حلقة لا نهائية فعندها المسلك ذو الأولوية العليا يجب أن ينتظر بشكل كبير حتى يحرر المسلك ذو الأولوية المنخفضة القفل، وهذا ما يُسمى بعكس الأولوية (أي أن الأولوية هنا لم يعد لها أهمية). ويُعد سقف الأولوية (`Priority Ceiling`) حلاً لهذه المشكلة، حيث أنه عندما يقوم أحد المسالك بقفل `Mutex` عندها سوف يتم رفع أولويته إلى أعلى من أعلى مسلك آخر بين المسلك ذي الأولوية المنخفضة والمسلك ذي الأولوية المرتفعة، وبالتالي هنا لم يعد يوجد مسلك ذو أولوية متوسطة.

**الشرح المبسّط:**
المشكلة تحدث بالترتيب التالي:
1. Thread منخفض الأولوية (L) يحجز `mutex`.
2. Thread عالي الأولوية (H) يحتاج نفس `mutex`، فينتظر.
3. Thread **متوسط** الأولوية (M) يصبح جاهزاً وليس بحاجة لـ `mutex` إطلاقاً، فيسحب المعالج من L (لأن أولويته أعلى من L).
4. إن كان M يحتوي حلقة طويلة، فإن L لا يحصل على المعالج ليكمل عمله ويحرر `mutex`، وبالتالي H ينتظر بشكل غير مباشر بسبب M — رغم أن أولوية H أعلى من M!

**حل سقف الأولوية (`Priority Ceiling`):** عند حجز أي `mutex`، تُرفع أولوية الـ Thread الحاجز مؤقتاً لتصبح **أعلى من أعلى أولوية محتملة** لأي Thread قد يطلب نفس المورد (وليس فقط أعلى من طالب واحد معروف)، مما يمنع أي Thread متوسط الأولوية من مقاطعته أثناء حجزه للمورد.

#### ⚖️ المقايضة: بدون حل مقابل Priority Ceiling

| | بدون حل (Priority Inversion يحدث) | مع Priority Ceiling |
| --- | --- | --- |
| المزايا | لا يوجد تعقيد إضافي في الجدولة | يمنع مقاطعة Thread متوسط الأولوية لحامل القفل |
| العيوب | قد ينتظر Thread عالي الأولوية لفترة غير متناسبة وغير متوقعة | رفع الأولوية المؤقت قد يُعقّد تحليل النظام الآني (real-time) |
| متى تختاره | أنظمة غير حساسة للتوقيت الصارم | أنظمة الوقت الحقيقي (real-time systems) الحرجة |

#### 🤔 تفعيل الفهم (اسأل نفسك):
> **سؤال:** هل عكس الأولوية هو استعصاء (Deadlock)؟
> **لماذا هذا مهم؟** لا، لأن L في النهاية **سيحصل على المعالج** ويحرر القفل عاجلاً أم آجلاً (بمجرد انتهاء M)، فالنظام لا يتجمد نهائياً، لكن التأخير قد يكون غير مقبول في أنظمة الوقت الحقيقي — لذا يُصنَّف كمشكلة جدولة لا كاستعصاء دائم.

### 9. مسألة عشاء الفلاسفة (Dining-Philosophers Problem)

**النص الأصلي يقول:** لدينا خمسة فلاسفة يقضون حياتهم بين الأكل والتفكير ولكنهم لا يستطيعون التفكير أثناء الأكل، كما أنهم لا يستطيعون الأكل أثناء التفكير. هؤلاء الفلاسفة يجلسون على طاولة مستديرة يوجد في منتصفها صحن من الرز. بجوار كل فيلسوف عصا واحدة فقط للأكل عن يمينه وعصا واحدة فقط للأكل عن يساره. الفيلسوف يستطيع استخدام العصا التي عن يمينه مباشرة أو عن يساره مباشرة فقط. من المستحيل تناول الرز بعصا واحدة بعصا واحدة أي أن أي فيلسوف يحتاج لعصاتين ليتمكن من الأكل. عندما ينتهي من الأكل يضع العصي على أمكنتها على الطاولة سامحاً بذلك لفيلسوف آخر مجاور أن يلتقط إحداها (المجاورة لهذا الأخير) ثم يعود للتفكير من جديد. الفلاسفة لا تتحدث مع بعضها أثناء الجلوس.

**الشرح المبسّط:**
هذه مسألة تمثيلية شهيرة لمشكلة **التنافس على موارد متعددة مشتركة بترتيب دائري** (5 فلاسفة، 5 عصي، كل فيلسوف يحتاج عصاتين متجاورتين). المشكلة الجوهرية: إن حاول **كل** الفلاسفة الخمسة التقاط العصا اليمنى في نفس اللحظة، فسيحمل كل منهم عصا واحدة فقط وينتظر العصا اليسرى إلى الأبد — وهذا **استعصاء كلاسيكي بترتيب دائري (`circular wait`)** بين 5 عمليات و5 موارد.

**💡 التشبيه:**
> يشبه هذا 5 أشخاص حول طاولة دائرية، كل منهم يحتاج شوكة وسكيناً لتناول الطعام، لكن يوجد فقط 5 أدوات موزعة بينهم (أداة واحدة بين كل شخصين)؛ إن التقط الجميع الأداة الموجودة على يمينهم في نفس اللحظة، لن يستطيع أحد الحصول على الأداة الثانية.
> **وجه الشبه:** العصا الواحدة المشتركة بين فيلسوفين = مورد يتنافس عليه طرفان متجاوران.

### 10. حل مسألة عشاء الفلاسفة بالمسيّرات (الحل الساذج وخطره)

**النص الأصلي يقول:** نستخدم مصفوفة من المسيّرات الثنائية `chopstick[5]` تأخذ جميعها القيمة الابتدائية 1. ويكون الفيلسوف رقم `(i)`:
```
while (true) {
    wait(chopStick[i]);              // get left chopstick
    wait(chopStick[(i + 1) % 5]);    // get right chopstick
    // eat for a while
    signal(chopStick[i]);            // return left chopstick
    signal(chopStick[(i + 1) % 5]);  // return right chopstick
    // think for awhile
}
```

**الشرح المبسّط:**
كل عصا هي مسيّر ثنائي (`chopstick[i]`) مُهيَّأ إلى 1 (متاحة). الفيلسوف `i` يحاول حجز عصاه اليسرى `chopstick[i]` ثم اليمنى `chopstick[(i+1)%5]`. **هذا الحل معطوب** رغم صحته المنطقية الظاهرية: إن نفّذ كل الفلاسفة الخمسة `wait(chopStick[i])` (حجز العصا اليسرى) في نفس اللحظة تقريباً، ستنجح جميع محاولات الحجز الأولى (كل عصا تُحجز مرة واحدة فقط بنجاح لأنها ملك لفيلسوف واحد فقط كعصا يسرى له)، لكن بعدها **يحاول الجميع حجز العصا اليمنى وهي محجوزة أصلاً** من الفيلسوف المجاور — فيقع الجميع في استعصاء دائري تام.

```c
// Philosopher i (naive/buggy version)
while (true) {
    wait(chopStick[i]);              // get left chopstick
    wait(chopStick[(i + 1) % 5]);    // get right chopstick  <-- DEADLOCK RISK HERE
    eat();
    signal(chopStick[i]);            // return left chopstick
    signal(chopStick[(i + 1) % 5]);  // return right chopstick
    think();
}
```

#### شرح كل سطر:
1. `wait(chopStick[i])` → حجز العصا اليسرى؛ تنجح دائماً لأول محاولة لكل فيلسوف بشكل مستقل.
2. `wait(chopStick[(i+1)%5])` → حجز العصا اليمنى؛ هذه النقطة الخطرة: إن حجز كل فيلسوف عصاه اليسرى أولاً، ستكون كل عصا يمنى محجوزة من الجار، فيحدث انتظار دائري.

**الناتج المتوقع:**
> في أسوأ الحالات (تزامن تام لكل الفلاسفة)، يتوقف النظام بالكامل ولا يأكل أي فيلسوف أبداً — استعصاء تام.

#### الفهم الخاطئ الشائع ❌: "طالما أن كل عصا مسيّر ثنائي منفصل، فالحل صحيح تماماً."
#### الفهم الصحيح ✅: صحة `wait`/`signal` الفردية لا تكفي؛ يجب أيضاً تفادي **الترتيب الدائري في حجز موارد متعددة**، وإلا يحدث استعصاء رغم أن كل مسيّر يعمل بشكل صحيح على حدة.

#### حلول شائعة لمنع هذا الاستعصاء (خارج النص الحرفي — شرح زيادة للفهم):
> من الحلول الشائعة: 1) السماح بأربعة فلاسفة كحد أقصى بالجلوس للأكل في نفس الوقت (مسيّر عداد إضافي). 2) جعل فيلسوف واحد فقط (كسر التناظر) يحجز العصا اليمنى أولاً ثم اليسرى بعكس الباقي. 3) استخدام حل مبني على `Monitor` (كما سيُشرح لاحقاً) يتحقق من حالة الجارين قبل السماح بالأكل.

### 11. مشاكل المسيّرات (Semaphore Usage Problems)

**النص الأصلي يقول:** يتطلب الاعتماد على المسيّرات كأداة للمزامنة حذراً في استخدام العمليتين `wait()` و`signal()` بحيث يتم استدعاء هذه العمليات بالشكل الصحيح. هناك العديد من الأخطاء الشائعة في التعامل مع هاتين العمليتين الأمر الذي يتسبب بفشل آلية المزامنة وبالتالي عدم القدرة على حل مشكلة المناطق الحرجة وتحقيق الاقصاء التبادلي. من أكثر هذه الأخطاء شيوعاً التالية: `Signal(mutex) ..... Wait(mutex)` — يتسبب هذا الاستخدام الخاطئ بفشل الاقصاء التبادلي نتيجة لتمكن جميع العمليات من الدخول إلى مناطقها الحرجة. `Wait(mutex) ..... Wait(mutex)` — يتسبب هذا الاستخدام الخاطئ بحدوث استعصاء. حذف العملية `Wait(mutex)` مما يؤدي إلى فشل الاقصاء التبادلي، أو حذف العملية `Signal(mutex)` مما يؤدي إلى استعصاء.

**الشرح المبسّط:**
هذه قائمة بأشهر **أربعة أخطاء** يقع فيها المبرمجون عند استخدام `mutex`:

| الخطأ | الأثر |
| --- | --- |
| عكس الترتيب: `signal` قبل `wait` | فشل الإقصاء التبادلي تماماً — كل العمليات تدخل بحرية دون أي حماية |
| تكرار `wait` مرتين متتاليتين بدون `signal` بينهما | استعصاء فوري — العملية نفسها تحجب نفسها عند ثاني `wait` |
| نسيان `wait(mutex)` بالكامل | فشل الإقصاء التبادلي — لا حماية إطلاقاً للمنطقة الحرجة |
| نسيان `signal(mutex)` بالكامل | استعصاء دائم لكل العمليات الأخرى المنتظرة على نفس المسيّر |

**💡 التشبيه:**
> يشبه نسيان `signal(mutex)` نسيان إعادة مفتاح غرفة اجتماعات مشتركة بعد الاستخدام — لن يستطيع أي شخص آخر دخول الغرفة أبداً حتى لو كانت فارغة فعلياً.
> **وجه الشبه:** المفتاح المفقود = القيمة المنقوصة من `mutex` التي لم تُعَد أبداً.

#### مهم للامتحان ⚠️:
> هذه الأخطاء الأربعة (عكس الترتيب، تكرار wait، حذف wait، حذف signal) هي **أكثر أسئلة "اكتشف الخطأ" شيوعاً** في امتحانات هذه المادة — احفظ الأثر الدقيق لكل واحد منها (فشل إقصاء تبادلي **أو** استعصاء).

### 12. حل مشكلة الحلاق النائم (Sleeping Barber Problem)

**النص الأصلي يقول:** إذا افترضنا أن لدينا صالون حلاقة به حلاق واحد وكرسي حلاقة واحد وعدد من الكراسي للانتظار. إذا لم يكن هنالك زبائن سيجلس الحلاق في كرسي الحلاقة وينام، وعندما يصل زبون فسيقوم بإيقاظ الحلاق. إذا جاء زبون آخر وكان الحلاق يحلق للزبون الأول فسيجلس الزبون الثاني على أحد كراسي الانتظار. كلما يصل زبون سيجلس في كرسي انتظار إلى أن تمتلي كراسي الانتظار. إذا جاء زبون ووجد كراسي الانتظار مشغولة فعليه مغادرة الصالون.

**الشرح المبسّط:**
هذه مسألة تزامن أخرى كلاسيكية تُمثّل **علاقة منتج/مستهلك مع حجم مخزن محدود (كراسي الانتظار) وحالة إضافية (نوم/يقظة الحلاق)**. الزبائن هم "المنتجون" لطلبات الحلاقة، والحلاق هو "المستهلك" الوحيد. إن لم يوجد زبائن ينام الحلاق (توفير موارد بدل الانتظار الشاغل)، وإن امتلأت كراسي الانتظار يُرفض أي زبون جديد (مماثل لمخزن ممتلئ في Producer-Consumer، لكن مع خيار "رفض" بدل الانتظار).

**💡 التشبيه:**
> النص الأصلي نفسه يشرح الفكرة بمثال حي: صالون حلاقة واقعي — الحلاق ينام إن لم يوجد عمل، ويستيقظ فور وصول زبون، والزبائن الذين يصلون أثناء انشغاله ينتظرون على كراسي محدودة العدد، وإن امتلأت يغادرون.

#### لحل هذه المشكلة نستخدم ثلاث مسيّرات:
- مسيّر للزبائن المنتظرين (`Customers`).
- مسيّر للحلاق (`Barber`) (لنعرف هل هو نائم (عاطل) أم يعمل).
- مسيّر لتحقيق الإقصاء التبادلي (`mutex` / `accessSeats`)؛ بحيث لا نسمح لشخصين بالحلاقة في وقت واحد.

كل ما يحضر زبون سيحاول الحصول على كرسي الحلاقة (`mutex`) وسيظل كذلك حتى ينجح. على الزبون الذي يحضر للصالون أن يحسب عدد الزبائن المنتظرين فإذا كان أقل من عدد الكراسي فسيجلس وإلا فسيغادر (يحاول الحصول على كرسي سواء في غرفة الانتظار أو كرسي الحلاق نفسه).

إذا وجد الزبون كرسي سيجلس وينقص عدد الكراسي الفارغة بواحد (`critical section`).

يقوم الزبون بإرسال إشارة إلى الحلاق لإيقاظه، وسيحرر `mutex` للسماح للزبائن (أو الحلاق) بالمقدرة على الحصول عليه. إذا كان الحلاق مشغولاً فعلياً بالزبائن الانتظار. سيجلس الحلاق في انتظار دائم، يتم إيقاظه بأي زبون من المنتظرين، وعندما يستيقظ سيرسل إشارات للزبائن المنتظرين بواسطة السيمافور للسماح لهم بالحلاقة، واحداً كل مرة. هذه المشكلة تحتوي على حلاق واحد لذلك تسمى أحياناً (`single sleeping barber problem`). فيما يلي شفرات مبدئية (`pseudo-code`)، تضمن التزامن بين الحلاق والزبائن خالية من الاختناق، ولكنها قد تقود إلى حرمان الزبون. نجد `P` و`V` هما دالتين توفرهما السيمافورات.

```
Semaphore Customers = 0
Semaphore Barber = 0
Semaphore accessSeats (mutex) = 1
```

**الشرح المبسّط:**
- `Customers` (Counting Semaphore = 0): يعدّ عدد الزبائن المنتظرين حالياً؛ الحلاق ينام عبر `wait(Customers)` إن كانت القيمة 0.
- `Barber` (Counting Semaphore = 0): يُخبر الزبون أن الحلاق جاهز فعلياً لحلاقته (وليس فقط "مستيقظاً" بل جاهزاً لهذا الزبون تحديداً).
- `accessSeats` / `mutex` (Binary Semaphore = 1): يحمي المتغير المشترك `waiting` (عدد الكراسي المشغولة) من التداخل بين زبائن متعددين يصلون في نفس اللحظة.

#### مهم للامتحان ⚠️:
> النص يذكر صراحة أن هذا الحل **قد يقود إلى حرمان الزبون (Starvation)** رغم أنه خالٍ من الاستعصاء — فرق مهم يُختبر كثيراً: **عدم وجود Deadlock لا يعني عدم وجود Starvation**.

### 13. الكود الكامل لحل الحلاق النائم

**النص الأصلي يقول:**
```c
#define CHAIRS 5                    /* number of chairs for waiting customers */
typedef int semaphore;
semaphore customers = 0;            /* number of waiting customers */
semaphore barbers = 0;              /* number of barbers waiting for customers */
semaphore mutex = 1;                /* for mutual exclusion */
int waiting = 0;                    /* customers are waiting, not being haircut */

void Barber(void) {
    while (TRUE) {
        down(customers);            /* go to sleep if number of customers is 0 */
        down(mutex);                 /* acquire access to 'waiting' */
        waiting = waiting - 1;        /* decrement count of waiting customers */
        up(barbers);                   /* one barber is now ready to cut hair */
        up(mutex);                       /* release 'waiting' */
        cut_hair();                        /* cut hair, non-CS */
    }
}

void customer(void) {
    down(mutex);                     /* enter CS */
    if (waiting < CHAIRS) {
        waiting = waiting + 1;         /* increment count of waiting customers */
        up(customers);                   /* wake up barber if necessary */
        up(mutex);                         /* release access to 'waiting' */
        down(barbers);                       /* wait if no free barbers */
        get_haircut();                         /* non-CS */
    } else {
        up(mutex);                       /* shop is full, do not wait */
    }
}
```

**الشرح المبسّط:**
`down()` هنا هي نفسها `wait()` و`up()` هي نفسها `signal()` (تسمية بديلة شائعة). الكود يطبّق تماماً ما شُرح سابقاً:
- الحلاق ينام على `down(customers)` إن لم يوجد زبائن.
- الزبون يفحص `waiting < CHAIRS` بأمان داخل `mutex`؛ إن وُجد مكان يزيد العداد ويوقظ الحلاق (`up(customers)`) ثم ينتظر جاهزية الحلاق تحديداً (`down(barbers)`) قبل الحلاقة الفعلية.
- إن كان الصالون ممتلئاً (`waiting >= CHAIRS`)، يُحرر الزبون `mutex` فوراً ويغادر دون انتظار.

#### شرح كل سطر (Barber):
1. `down(customers);` → ينام إن لم يوجد زبائن على الإطلاق (القيمة 0).
2. `down(mutex); waiting = waiting - 1; up(mutex);` → إنقاص عداد الانتظار بأمان (لأن زبوناً سيبدأ حلاقته الآن، فلم يعد "منتظراً").
3. `up(barbers);` → إخبار الزبون التالي في الرتل بأن الحلاق جاهز له تحديداً.
4. `cut_hair();` → تنفيذ الحلاقة الفعلية خارج أي منطقة حرجة.

#### شرح كل سطر (Customer):
1. `down(mutex);` → دخول آمن لفحص/تعديل `waiting`.
2. `if (waiting < CHAIRS) {...} else { up(mutex); }` → إما الانضمام لصف الانتظار أو المغادرة الفورية بعد تحرير القفل.
3. `up(customers);` → توقظ الحلاق إن كان نائماً (أو تزيد العداد إن كان مشغولاً بالفعل، لينتظر دوره).
4. `down(barbers);` → الزبون ينتظر تحديداً أن يصبح الحلاق جاهزاً له (وليس فقط مستيقظاً بشكل عام).

**الناتج المتوقع:**
> يعمل الحلاق دون انتظار شاغل، ولا يدخل زبونان لكرسي الحلاقة في نفس اللحظة، ولا يتجاوز عدد المنتظرين سعة الكراسي؛ لكن -كما ذُكر- قد يحدث حرمان لبعض الزبائن نظرياً حسب سياسة الرتل الداخلي لكل مسيّر.

### 14. المراقبات (Monitors)

**النص الأصلي يقول:** لجعل كتابة البرامج الصحيحة أسهل، تم تقديم أداة عالية المستوى لتحقيق المزامنة تسمى المراقب (`Monitor`). المراقب هو عبارة عن مجموعة إجراءات ومتحولات وبنى معطيات مجمّعة بنوع خاص من الوحدات النمطية (`Classes`)، بحيث تستطيع العمليات استدعاء الإجراءات الموجودة في المراقب لكنها لا تستطيع الوصول مباشرة إلى البيانات الداخلية في المراقب انطلاقاً من إجراءات لا تنتمي إلى المراقب.
```
monitor monitor-name {
    // shared variable declarations
    procedure P1 (...) { .... }
    ...
    procedure Pn (...) {......}
    Initialization code (....) { ... }
}
```

**الشرح المبسّط:**
`Monitor` هو **تجريد أعلى مستوى** من `Semaphore`: بدل أن يتذكر المبرمج بنفسه أين يضع `wait`/`signal` (وهو عرضة للأخطاء الأربعة المذكورة سابقاً)، فإن `Monitor` **يُخفي** كل هذه التفاصيل داخل بنية شبيهة بالـ `Class`: البيانات المشتركة خاصة (`private`) ولا يمكن الوصول إليها إلا عبر إجراءات (`procedures`) معرّفة داخل المراقب نفسه، والمترجم (`Compiler`) هو من يضمن الإقصاء التبادلي تلقائياً، وليس المبرمج.

**💡 التشبيه:**
> يشبه `Monitor` خزنة بنكية آلية: لا يمكنك الوصول للنقود مباشرة، بل فقط عبر إجراءات محددة (سحب، إيداع) يديرها النظام نفسه ويضمن ألا يدخل شخصان لنفس الخزنة في نفس اللحظة — دون أن تحتاج أنت لقفل الباب يدوياً بنفسك.
> **وجه الشبه:** إجراءات المراقب (procedures) = عمليات السحب/الإيداع المُدارة تلقائياً؛ البيانات الداخلية الخاصة = محتوى الخزنة غير القابل للوصول المباشر.

### 15. خاصية الإقصاء التبادلي التلقائي في المراقبات

**النص الأصلي يقول:** للمراقبات خاصية هامة تجعلها مفيدة لتحقيق الإقصاء التبادلي، وهي أن عملية واحدة فقط تستطيع أن تكون نشطة في المراقب في أي لحظة زمنية، بحيث يتولى المترجم (`Compiler`) تحقيق ذلك وليس المبرمج. عندما تستدعي عملية إجراء مراقب، فإنه سيتم فحص ما إذا كانت أي عملية أخرى نشطة داخل المراقب بحيث يجب على العملية المستدعية الانتظار ريثما تخرج العملية الأخرى من المراقب وعندها يسمح للعملية المستدعية بالدخول.

**الشرح المبسّط:**
هذه هي **الميزة الجوهرية** لـ `Monitor`: الإقصاء التبادلي **مضمون تلقائياً وبنيوياً** بحكم تعريف المراقب نفسه — لا حاجة لكتابة `wait(mutex)`/`signal(mutex)` يدوياً، فالمترجم يُدرج آلية `entry queue` تلقائياً تمنع أكثر من عملية واحدة من تنفيذ أي إجراء داخل المراقب في نفس اللحظة.

#### 📊 المخطط: بنية المراقب مع رتل الدخول (Entry Queue)

#### ما هذا المخطط؟
> يوضّح كيف تصطف العمليات الراغبة باستدعاء أي إجراء من إجراءات المراقب في `entry queue` خارجي، بينما تُنفَّذ العمليات المسموح لها بالدخول واحدة تلو الأخرى فقط على البيانات المشتركة الداخلية.

#### وصف العُقد:

| --- | --- | --- | --- |
| 1 | `entry queue` | رتل خارجي | العمليات المنتظرة لدخول المراقب |
| 2 | `shared data` | بيانات خاصة | لا يمكن الوصول إليها إلا عبر الإجراءات |
| 3 | `operations (procedures)` | إجراءات المراقب | الوسيلة الوحيدة للتفاعل مع البيانات المشتركة |
| 4 | `initialization code` | كود تهيئة | يُنفَّذ مرة واحدة عند إنشاء المراقب |

### 16. الحاجة لمتحولات شرطية داخل المراقب (Condition Variables)

**النص الأصلي يقول:** على الرغم من أن المراقبات تقدم طريقة سهلة لتحقيق الإقصاء التبادلي، فإن ذلك ليس كافياً، لأننا نحتاج أيضاً إلى إيقاف العمليات عندما لا تستطيع متابعة التنفيذ، كما في مسألة المنتج-مستهلك مثلاً. يكمن الحل في تقديم متحولات شرطية (`Condition Variables`) مع العمليتين `wait()` و`signal()` على هذه المتحولات. عندما يكتشف إجراء ضمن المراقب أنه لا يستطيع المتابعة فإنه ينفذ `wait()` على أحد متحولات الشرط، مما يؤدي إلى إيقاف العملية المستدعية كما يسمح لعملية أخرى بالدخول إلى المراقب. تستطيع عملية ما إيقاظ عملية ثانية بتنفيذ العملية `signal()` على متحول الشرط الذي تنتظره تلك العملية.

**الشرح المبسّط:**
الإقصاء التبادلي وحده لا يكفي لحل مسائل مثل `Producer-Consumer` (نحتاج أيضاً "انتظاراً حتى يتحقق شرط ما"). لذلك أضيفت `Condition Variables` مع عمليتين إضافيتين:
- `wait()` على متحول شرط: تُوقف العملية الحالية **وتسمح فوراً لعملية أخرى بدخول المراقب** (بخلاف `wait(S)` في المسيّر العادي الذي لا يُشترط أن يفرغ المكان لغيره تلقائياً).
- `signal()` على متحول شرط: توقظ عملية أخرى كانت تنتظر تحديداً على هذا الشرط.

**💡 التشبيه:**
> يشبه هذا موظف استقبال (المراقب) يستقبل زبوناً واحداً في كل مرة (إقصاء تبادلي)، لكن أحياناً الزبون يحتاج انتظار توفر خدمة معينة (شرط)؛ فيخرج مؤقتاً لغرفة انتظار خاصة بهذه الخدمة تحديداً (`condition variable`) دون أن يمنع باقي الزبائن من دخول المكتب أثناء انتظاره.
> **وجه الشبه:** غرفة الانتظار الخاصة بالخدمة = قائمة الانتظار المرتبطة بمتحول الشرط تحديداً.

### 17. أثر signal() في المراقب عند غياب عملية منتظرة على المتحول الشرطي

**النص الأصلي يقول:** متحولات الشرط ليس لها عدادات لأنها لا تراكم الإشارات من أجل الاستخدام اللاحق كما تفعل المسيّرات، وبالتالي إذا تم تأشير (`Signal()`) متحول شرط دون وجود عملية تنتظر عنده فإن الإشارة ستضيع للأبد.

**الشرح المبسّط:**
هذا فرق **جوهري** بين `Semaphore` و`Condition Variable`: المسيّر يحتفظ بقيمة عددية دائمة (يمكن أن تصبح موجبة حتى بدون منتظرين، فتُستهلك لاحقاً)، أما متحول الشرط **لا يخزن أي قيمة إطلاقاً** — فإن استُدعي `signal()` ولم تكن هناك عملية منتظرة فعلياً على ذلك الشرط في تلك اللحظة، تضيع الإشارة تماماً بلا أي أثر لاحق (شبيه بمشكلة `Lost Wakeup` الأصلية، لكنه هنا سلوك **متعمد ومعروف** ضمن تصميم `Monitor` وليس خطأً بنيوياً).

#### مهم للامتحان ⚠️:
> عكس `Semaphore` تماماً: المسيّر "يتذكر" الإشارات المرسلة (عبر قيمته العددية)، بينما متحول الشرط **لا يتذكر شيئاً** — لذا يجب دائماً التأكد من وجود عملية منتظرة فعلياً وقت استدعاء `signal()` على متحول شرط، وإلا فالإشارة بلا فائدة.

### 18. حل مسألة عشاء الفلاسفة باستخدام المراقبات

**النص الأصلي يقول:** يجب على كل فيلسوف أن يستدعي العمليتين `pickup()` و`putdown()` وفق التسلسل التالي:
```
DiningPhilosophters.pickup(i);
    EAT
DiningPhilosophers.putdown(i);
```
```
monitor DP {
    enum {THINKING, HUNGRY, EATING} state[5];
    condition self[5];

    initialization_code() {
        for (int i = 0; i < 5; i++)
            state[i] = THINKING;
    }

    void test(int i) {
        if ((state[(i + 4) % 5] != EATING) &&
            (state[i] == HUNGRY) &&
            (state[(i + 1) % 5] != EATING)) {
            state[i] = EATING;
            self[i].signal();
        }
    }

    void pickup(int i) {
        state[i] = HUNGRY;
        test(i);
        if (state[i] != EATING) self[i].wait;
    }

    void putdown(int i) {
        state[i] = THINKING;
        // test left and right neighbors
        test((i + 4) % 5);
        test((i + 1) % 5);
    }
}
```

**الشرح المبسّط:**
هذا الحل **يتفادى تماماً** مشكلة الاستعصاء التي وقع فيها الحل الساذج بالمسيّرات المباشرة، لأنه لا يحجز العصي بشكل منفصل، بل يفحص **حالة الجارين** قبل السماح بالأكل:
- كل فيلسوف له حالة من ثلاث: `THINKING`، `HUNGRY` (جائع وينتظر عصيّاً)، `EATING`.
- دالة `test(i)`: تفحص هل الفيلسوف `i` جائع، وهل جاراه (اليسار `(i+4)%5` واليمين `(i+1)%5`) **ليسا** في حالة `EATING` حالياً؛ إن تحقق كل ذلك، يتحول الفيلسوف `i` لحالة `EATING` فوراً ويُوقظ (`signal`) إن كان بانتظار.
- `pickup(i)`: يُعلن الفيلسوف جوعه، يستدعي `test(i)` مباشرة (فقد يستطيع الأكل فوراً إن كان الجاران غير مشغولين)، وإن لم ينجح ينتظر (`self[i].wait`) على متحول شرطه الخاص.
- `putdown(i)`: عند الانتهاء يعود الفيلسوف للتفكير، ثم يفحص **جاريه تحديداً** (وليس نفسه) عبر استدعاء `test` عليهما، لأنهما ربما كانا ينتظران عصياً كان الفيلسوف `i` يحجزها ضمنياً بحالته.

```algorithm
1 | إعلان الجوع | pickup(i): state[i]=HUNGRY | الفيلسوف يعلن رغبته بالأكل
2 | فحص إمكانية الأكل فوراً | test(i) | يتحقق إن كان الجاران غير EATING حالياً
3 | انتظار محتمل | if(state[i]!=EATING) self[i].wait | ينتظر فقط إن تعذر الأكل فوراً
4 | إنهاء الأكل | putdown(i): state[i]=THINKING | يعلن انتهاءه من الأكل
5 | تحرير الفرصة للجيران | test((i+4)%5); test((i+1)%5) | يفحص إمكانية أكل الجارين الآن بعد تحريره لموارده ضمنياً
```

#### نقاط التنفيذ:
- **لا استعصاء إطلاقاً** في هذا الحل، لأن أي فيلسوف لا يحجز مورداً جزئياً أبداً (إما يأكل بكلا "العصاتين" منطقياً عبر الحالة، أو لا يأكل إطلاقاً وينتظر)، فلا يوجد سيناريو "امتلك نصف الموارد وانتظر النصف الآخر".

#### مهم للامتحان ⚠️:
> لاحظ أن `putdown(i)` تفحص **الجيران** (`i-1` و`i+1`) وليس الفيلسوف `i` نفسه، لأن الفيلسوف `i` أصلاً عاد للتفكير ولا يحتاج لفحص حالته؛ الهدف هو تحرير الفرصة لمن كان ينتظر بسببه.

### 19. تطبيق عملي للمراقب: BoundedBuffer بلغة Java

**النص الأصلي يقول:** (كود عملي يوضّح تطبيق مفهوم `Monitor` باستخدام الكلمة المفتاحية `synchronized` في Java، حيث الكائن `BoundedBuffer` هو المراقب نفسه، ويستخدم `wait()`/`notifyAll()` المدمجتين في Java كبديل عملي لـ `condition variables`).

**الشرح المبسّط:**
> **(شرح زيادة للفهم):** رغم أن هذا الكود مكتوب بلغة برمجة فعلية (Java)، إلا أن الغرض التعليمي منه هو **ربط المفهوم النظري لـ Monitor بتطبيق عملي حقيقي** يستخدمه المبرمجون فعلاً. الكلمة `synchronized` أمام دالة تجعل الكائن بأكمله يتصرف كمراقب: عملية واحدة (Thread) فقط تنفذ أي دالة `synchronized` من هذا الكائن في أي لحظة، تماماً كخاصية الإقصاء التبادلي التلقائي المذكورة في التعريف النظري لـ Monitor. الدالتان `wait()` و`notifyAll()` المدمجتان في اللغة تُطبّقان بالضبط مفهومي `wait`/`signal` على متحول شرط ضمني واحد للكائن.

```java
// This object itself acts as a Monitor
class BoundedBuffer {
    private final Queue<Integer> buffer = new LinkedList<>();
    private final int capacity;

    public BoundedBuffer(int capacity) {
        this.capacity = capacity;              // set buffer capacity
    }

    public synchronized void produce(int item) throws InterruptedException {
        while (buffer.size() == capacity) {     // if buffer full, wait (condition variable)
            wait();
        }
        buffer.add(item);                        // add item to buffer
        notifyAll();                               // wake up waiting consumers
    }

    public synchronized int consume() throws InterruptedException {
        while (buffer.isEmpty()) {               // if buffer empty, wait (condition variable)
            wait();
        }
        int item = buffer.poll();                 // remove item from buffer
        notifyAll();                                // wake up waiting producers
        return item;
    }
}
```

#### شرح كل سطر:
1. `public synchronized void produce(...)` → الدالة بأكملها منطقة حرجة محمية تلقائياً بحكم `synchronized`؛ لا حاجة لـ `wait(mutex)`/`signal(mutex)` يدوياً.
2. `while (buffer.size() == capacity) { wait(); }` → استخدام `while` لا `if` عمداً (نقطة تصميمية مهمة، انظر التحذير أدناه)؛ ينتظر ضمن متحول الشرط الضمني للكائن.
3. `buffer.add(item); notifyAll();` → إضافة العنصر ثم إيقاظ **جميع** الخيوط المنتظرة (منتجين ومستهلكين معاً)، لأن Java لا تفرّق افتراضياً بين نوعي الانتظار في متحول شرط واحد ضمني.
4. الشيء نفسه بالتماثل في `consume()`.

**الناتج المتوقع:**
> يعمل هذا الحل بشكل صحيح كتطبيق عملي لمبدأ Monitor، وإن كان أقل دقة من استخدام متحولي شرط منفصلين (`notFull`/`notEmpty`) كما في المثال التالي، بسبب استخدام `notifyAll()` العام الذي قد يوقظ خيوطاً غير معنية فتعيد فحص شرطها والعودة للنوم فوراً (كلفة أداء إضافية طفيفة).

#### مهم للامتحان ⚠️:
> استخدام `while` بدل `if` عند انتظار شرط داخل Monitor هو **قاعدة إلزامية**: لأن `notifyAll()` قد يوقظ خيطاً حتى لو لم يتحقق شرطه بعد فعلياً (بسبب تسابق خيوط أخرى استهلكت الفرصة قبله)؛ فإعادة فحص الشرط بعد الاستيقاظ (`while`) ضرورية لتفادي التصرف بناءً على افتراض خاطئ.

### 20. تطبيق متقدم: استخدام Condition منفصلين (notFull / notEmpty) مع ReentrantLock

**النص الأصلي يقول:** (كود متقدم يستخدم `java.util.concurrent.locks.Lock` و`ReentrantLock` و`Condition` صراحة، حيث يُعرَّف قفل صريح (`lock`) ومتحولا شرط مستقلان (`notFull` و`notEmpty`) بدل الاعتماد على القفل الضمني الواحد للكائن).

**الشرح المبسّط:**
> **(شرح زيادة للفهم):** هذا تطبيق **أكثر دقة ومطابقة** للنموذج النظري الأصلي للمراقب مع متحولات شرط متعددة (كما في مثال عشاء الفلاسفة النظري الذي استخدم `condition self[5]` منفصلة لكل فيلسوف). بدل الاعتماد على قفل ضمني واحد (`synchronized`)، يُعرَّف `lock` صريح، ثم متحولا شرط منفصلان تماماً: `notFull` يخص المنتجين فقط، و`notEmpty` يخص المستهلكين فقط. هذا يسمح باستخدام `signal()` بدلاً من `notifyAll()` (إيقاظ خيط واحد محدد بدل الجميع)، مما يحسّن الأداء بتقليل عمليات الإيقاظ غير الضرورية.

```java
class AdvancedMonitor {
    private final Queue<Integer> buffer = new LinkedList<>();
    private final int capacity;

    private final Lock lock = new ReentrantLock();          // explicit lock
    private final Condition notFull = lock.newCondition();    // condition for producers
    private final Condition notEmpty = lock.newCondition();    // condition for consumers

    public AdvancedMonitor(int capacity) {
        this.capacity = capacity;
    }

    public void produce(int item) throws InterruptedException {
        lock.lock();                                          // acquire lock manually
        try {
            while (buffer.size() == capacity) {                // wait in notFull queue
                notFull.await();
            }
            buffer.add(item);
            notEmpty.signal();                                  // wake ONE waiting consumer only
        } finally {
            lock.unlock();
        }
    }

    public int consume() throws InterruptedException {
        lock.lock();
        try {
            while (buffer.isEmpty()) {                          // wait in notEmpty queue
                notEmpty.await();
            }
            int item = buffer.poll();
            notFull.signal();                                     // wake ONE waiting producer only
            return item;
        } finally {
            lock.unlock();
        }
    }
}
```

#### شرح كل سطر:
1. `Lock lock = new ReentrantLock();` → قفل صريح يجب حجزه وتحريره يدوياً (بعكس `synchronized` الضمني).
2. `Condition notFull / notEmpty` → متحولا شرط منفصلان تماماً، كل واحد له قائمة انتظاره الخاصة بمعزل عن الآخر.
3. `lock.lock(); try {...} finally { lock.unlock(); }` → نمط إلزامي: التحرير داخل `finally` يضمن `signal`/`unlock` حتى في حال حدوث استثناء (يتفادى خطأ "نسيان signal" المذكور سابقاً في مشاكل المسيّرات).
4. `notFull.await()` / `notEmpty.signal()` → استدعاءات صريحة ودقيقة على متحول الشرط الصحيح فقط، لا على الكائن بأكمله.

**الناتج المتوقع:**
> أداء أفضل من `notifyAll()` العام، لأن كل `signal()` يوقظ خيطاً واحداً محدداً من القائمة الصحيحة فقط، دون إيقاظ خيوط غير معنية بالشرط الذي تحقق.

### 21. مقارنة Semaphore مقابل Monitor

**النص الأصلي يقول:** جدول مقارنة: وجه المقارنة | Semaphore | Monitor. المستوى: مستوى منخفض (صعب في البرمجة) | مستوى عالٍ (أسهل وأكثر أماناً). الإدارة: المبرمج مسؤول عن `wait()` و`signal()` | نظام التشغيل/المترجم يدير القفل تلقائياً. الأخطاء: خطأ واحد في الترتيب يؤدي لـ `Deadlock` | أقل عرضة للأخطاء البرمجية المنطقية.

**الشرح المبسّط:**
هذا الجدول يلخّص كل ما سبق شرحه بشكل مقارن مباشر: `Semaphore` أداة تقنية دقيقة لكنها **يدوية بالكامل** وعرضة لأخطاء الترتيب (كما رأينا في قسم "مشاكل المسيّرات")، بينما `Monitor` أداة **مُدارة تلقائياً** من قبل بيئة التنفيذ (المترجم أو نظام التشغيل)، فتقلل احتمال الوقوع في تلك الأخطاء الشائعة، على حساب مرونة أقل قليلاً ومستوى تجريد أعلى قد يُخفي بعض التفاصيل الدقيقة عن المبرمج.

### ⚖️ المقايضة: Semaphore مقابل Monitor

| | Semaphore | Monitor |
| --- | --- | --- |
| المزايا | تحكم دقيق ومرن جداً، متوفر على مستوى النظام والعتاد | أسهل كتابة، أقل عرضة للأخطاء، الإقصاء التبادلي مضمون تلقائياً |
| العيوب | خطأ واحد بالترتيب (نسيان signal، عكس الترتيب) يؤدي لاستعصاء أو فشل إقصاء | أقل مرونة، يعتمد على دعم اللغة/بيئة التنفيذ له |
| متى تختاره | أنظمة منخفضة المستوى، تحكم دقيق بعدد موارد متعددة | تطوير تطبيقات عامة حيث الأمان والوضوح أولوية |

## الجزء الثاني: ملخص منظم شامل

### جدول التعريفات

| المصطلح | التعريف |
| --- | --- |
| `Bounded-Buffer Problem` | الاسم البديل لمسألة `Producer-Consumer` نظراً لمحدودية سعة المخزن |
| `Readers-Writers Problem` | مسألة تزامن بين نوعين من العمليات: قرّاء متعددون متوازون، وكاتب واحد حصري |
| `Deadlock` | حالة انتظار دائري بين عمليتين أو أكثر لا يمكن أن ينفك أبداً بدون تدخل خارجي |
| `Starvation` | دخول عملية في حالة انتظار لا نهائي دون أن تُنقل أبداً لحالة الجاهزية رغم عدم وجود استعصاء تقني |
| `Priority Inversion` | مشكلة جدولة تجعل عملية عالية الأولوية تنتظر بسبب عملية منخفضة الأولوية (وربما متوسطة تتدخل) |
| `Priority Ceiling` | حل لعكس الأولوية برفع أولوية حامل القفل مؤقتاً لأعلى من أي طالب محتمل له |
| `Dining Philosophers Problem` | مسألة تمثل استعصاءً محتملاً بترتيب دائري عند التنافس على موارد متعددة مشتركة |
| `Sleeping Barber Problem` | مسألة تزامن تمثل علاقة منتج/مستهلك بمخزن محدود (كراسي انتظار) مع حالة نوم/يقظة |
| `Monitor` | أداة مزامنة عالية المستوى تضمن الإقصاء التبادلي تلقائياً وتُخفي تفاصيل wait/signal |
| `Condition Variable` | متحول شرطي داخل المراقب، بلا عداد، يُستخدم مع `wait()`/`signal()` للانتظار حتى تحقق شرط |

### جدول المكونات (حل Producer-Consumer الكامل)

| المكوّن | الدور |
| --- | --- |
| `empty` (Counting Semaphore = N) | عدد الأماكن الفارغة المتبقية |
| `full` (Counting Semaphore = 0) | عدد العناصر الجاهزة للاستهلاك |
| `mutex` (Binary Semaphore = 1) | حماية عملية الإدراج/السحب من التداخل بين عمليات متعددة من نفس النوع |

### جدول المقارنات

| المعيار | Semaphore | Monitor |
| --- | --- | --- |
| مستوى التجريد | منخفض (يدوي بالكامل) | عالٍ (مُدار تلقائياً) |
| مسؤولية الإقصاء التبادلي | المبرمج (wait/signal يدوياً) | المترجم / بيئة التنفيذ |
| احتمال الخطأ البشري | مرتفع (4 أخطاء شائعة موثقة) | أقل، لكن غير معدوم (مثل نسيان `while` بدل `if`) |
| متحول العد | نعم (قيمة عددية تراكمية) | لا يوجد عداد لمتحول الشرط (الإشارة تضيع إن لم يوجد منتظر) |

| المعيار | Deadlock | Starvation |
| --- | --- | --- |
| هل ينتهي تلقائياً؟ | لا، أبداً بدون تدخل خارجي | قد ينتهي نظرياً لكن قد يطول لأجل غير مسمى عملياً |
| السبب النموذجي | انتظار دائري على موارد متعددة (ترتيب معكوس) | سياسة اختيار غير عادلة في رتل الانتظار |
| مثال من المحاضرة | `P0`/`P1` مع `wait(S)`/`wait(Q)` بترتيب معكوس | الكاتب في حل Readers-Writers عند تدفق قرّاء مستمر |

### مسرد المصطلحات (Glossary)

| المصطلح | المعنى المختصر |
| --- | --- |
| `readcount` | عدّاد عدد القرّاء النشطين حالياً في حل Readers-Writers |
| `wrt` | مسيّر يمثل قفل الكتابة الحصري في حل Readers-Writers |
| `chopstick[i]` | مصفوفة مسيّرات ثنائية تمثل العصي الخمس في مسألة الفلاسفة |
| `entry queue` | رتل الانتظار الخارجي لدخول المراقب |
| `down()` / `up()` | تسميتان بديلتان لـ `wait()` و`signal()` على التوالي |
| `synchronized` (Java) | كلمة مفتاحية تجعل الدالة/الكائن يتصرف كمراقب مع إقصاء تبادلي تلقائي |

### نقاط ذهبية 🔑

- حل `Producer-Consumer` الكامل يحتاج **ثلاثة** مسيّرات: `empty`، `full`، `mutex` — وليس مسيّراً واحداً.
- في `Readers-Writers`: القارئ الأول يحجز `wrt` نيابة عن الجميع، والقارئ الأخير يحرره؛ القرّاء بينهما لا يتعاملون مع `wrt` إطلاقاً.
- الاستعصاء ينتج غالباً من **ترتيب معكوس** في حجز موارد متعددة بين عمليتين أو أكثر.
- عدم وجود Deadlock لا يعني عدم وجود Starvation (كما في حلّي Readers-Writers والحلاق النائم).
- `Monitor` يضمن الإقصاء التبادلي تلقائياً، لكنه يحتاج `Condition Variables` إضافية للانتظار الشرطي (لأن الإقصاء وحده لا يكفي).
- متحول الشرط في Monitor **لا يملك عداداً**؛ أي `signal()` بلا منتظر فعلي تضيع تماماً.
- استخدم `while` دائماً لا `if` عند الانتظار على شرط داخل Monitor.

### أخطاء شائعة ⚠️

| الخطأ | التصحيح |
| --- | --- |
| عكس ترتيب `signal(mutex)` ثم `wait(mutex)` | يجب أن يكون `wait` دائماً قبل `signal` |
| تكرار `wait(mutex)` مرتين متتاليتين | استعصاء فوري؛ يجب `signal` بينهما إن كانت هناك حاجة فعلية |
| نسيان `wait(mutex)` | فشل إقصاء تبادلي تام |
| نسيان `signal(mutex)` | استعصاء دائم لكل المنتظرين |
| افتراض أن Monitor يضمن أيضاً الانتظار الشرطي تلقائياً | Monitor يضمن فقط الإقصاء التبادلي؛ الانتظار الشرطي يحتاج `Condition Variables` بشكل صريح |
| استخدام `if` بدل `while` عند فحص شرط الانتظار في Monitor | يجب استخدام `while` دائماً لإعادة الفحص بعد الاستيقاظ |

### خطوات وإجراءات المحاضرة

```algorithm
1 | انتظار مكان فارغ | Producer: wait(empty) | يُحجب إن كان المخزن ممتلئاً بالكامل
2 | دخول المنطقة الحرجة | Producer: wait(mutex) | حماية عملية الإدراج من تداخل منتج آخر
3 | إدراج العنصر | Producer: insert_item() | العملية الفعلية للإضافة
4 | خروج من المنطقة الحرجة | Producer: signal(mutex) | تحرير القفل فوراً
5 | إشارة عنصر جاهز | Producer: signal(full) | إخبار المستهلكين المحتملين
```

#### نقاط التنفيذ:
- الترتيب الصحيح إلزامي: `wait(empty)` أو `wait(full)` أولاً، ثم `wait(mutex)` — عكس هذا الترتيب قد يسبب استعصاء (يُشرح في تمرين لاحق).

```algorithm
1 | دخول قارئ وزيادة العداد | wait(mutex); readcount++; | بأمان تام
2 | حجز الكاتب عند الحاجة | if(readcount==1) wait(wrt); | فقط القارئ الأول يحجز
3 | تحرير mutex للسماح بالتوازي | signal(mutex); | يسمح لقرّاء آخرين بالانضمام فوراً
4 | القراءة الفعلية | reading is performed | متوازية بين كل القرّاء النشطين
5 | خروج قارئ وإنقاص العداد | wait(mutex); readcount--; | بأمان تام
6 | تحرير الكاتب عند الحاجة | if(readcount==0) signal(wrt); | فقط القارئ الأخير يحرر
7 | تحرير mutex | signal(mutex); | إنهاء دورة القارئ
```

#### نقاط التنفيذ:
- هذا الحل يُعطي أفضلية للقرّاء (`readers-priority`) وقد يُجوّع الكاتب إن استمر وصول قرّاء جدد بلا انقطاع.

```algorithm
1 | إعلان الجوع | pickup(i): state[i]=HUNGRY | إشارة نية الأكل
2 | فحص الجيران | test(i) | يفحص إن كان الجاران غير EATING
3 | الأكل أو الانتظار | state[i]=EATING أو self[i].wait | حسب نتيجة الفحص
4 | إنهاء الأكل | putdown(i): state[i]=THINKING | يعلن الانتهاء
5 | تحرير فرصة الجيران | test((i+4)%5); test((i+1)%5); | يفحص إمكانية أكل الجارين الآن
```

#### نقاط التنفيذ:
- هذا الحل بالمراقبات **خالٍ من الاستعصاء تماماً** بعكس الحل الساذج بالمسيّرات المنفصلة لكل عصا.

### أنماط الأكواد (Code Patterns)

| النمط | الاستخدام |
| --- | --- |
| `wait(empty); wait(mutex); ...; signal(mutex); signal(full);` | حل عام لمشكلة منتج بمخزن محدود مع حماية إضافية للمخزن نفسه |
| `wait(mutex); count++; if(count==1) wait(other); signal(mutex);` | نمط "أول واحد يحجز، آخر واحد يحرر" الشائع في Readers-Writers |
| `while (condition) { conditionVar.wait(); }` | نمط الانتظار الآمن على متحول شرط داخل Monitor (مع while لا if) |

### أنماط التعامل (Handling Patterns)

- عند تصميم حل يستخدم أكثر من مسيّر واحد، تحقق دائماً من **ترتيب الاستدعاء** (أيهما أولاً: العداد أم القفل؟) لأن الترتيب الخاطئ قد يُبطل الحل بالكامل أو يسبب استعصاء.
- عند مواجهة مسألة تنافس على موارد متعددة متجاورة (كعشاء الفلاسفة)، افحص أولاً إمكانية حدوث **ترتيب دائري (circular wait)** قبل قبول أي حل مبني على حجز منفصل لكل مورد.
- عند استخدام Monitor، تذكر دائماً أن الإقصاء التبادلي مضمون، لكن **الانتظار الشرطي مسؤوليتك** عبر `Condition Variables` صريحة.

### الأفكار الشاملة

المسائل الكلاسيكية (`Producer-Consumer`، `Readers-Writers`، `Dining Philosophers`، `Sleeping Barber`) ليست مجرد تمارين نظرية، بل هي **أنماط متكررة** تظهر في أنظمة حقيقية (قواعد بيانات، أنظمة تشغيل، خوادم شبكات)، وفهم حلولها الصحيحة (وأخطائها الشائعة) هو أساس تصميم أي نظام متزامن آمن؛ بينما `Monitor` يمثل تطور طبيعي لتبسيط هذه الحلول وتقليل مخاطرها البرمجية.

### السؤال 1
كم مسيّراً يحتاجها الحل الكامل لمسألة Producer-Consumer المذكور في المحاضرة؟
أ) مسيّر واحد فقط (mutex)
ب) مسيّران (empty و full فقط)
ج) ثلاثة مسيّرات (empty و full و mutex)
د) أربعة مسيّرات

**التعليل:**
- أ) خاطئ — mutex وحده لا يضبط عدد العناصر/الأماكن.
- ب) خاطئ — بدون mutex قد يتداخل منتجان أو مستهلكان في تعديل بنية المخزن.
- ج) ✅ صحيح تماماً، هذا هو الحل الرسمي المذكور حرفياً.
- د) خاطئ — لا حاجة لمسيّر رابع في هذا الحل.

### السؤال 2
ما القيمة الابتدائية لمسيّر `full` في حل Producer-Consumer؟
أ) N (سعة المخزن)
ب) 1
ج) 0
د) غير محدد

**التعليل:** ج) ✅ لأنه لا يوجد أي عنصر جاهز عند البداية. أ) هذه قيمة `empty`. ب) هذه قيمة `mutex`. د) القيمة محددة صراحة في النص.

### السؤال 3

ب) القارئ الأول فقط (عندما readcount يصبح 1)
ج) الكاتب فقط عند القراءة
د) لا أحد يستدعيها

**التعليل:** ب) ✅ مطابق حرفياً للكود المعروض. أ) لو فعل كل قارئ ذلك لمُنع التوازي بين القرّاء. ج) الكاتب يستدعي `wait(wrt)` أيضاً لكن السؤال يخص القرّاء تحديداً وحصراً على الأول منهم. د) خاطئ تماماً.

أ) القرّاء
ب) الكاتب
ج) كلاهما بنفس الاحتمال
د) لا توجد مشكلة Starvation في هذا الحل

أ) لا فرق، مترادفان تماماً
ب) Deadlock انتظار دائري لا ينفك أبداً، بينما Starvation انتظار طويل قد ينتهي نظرياً لكن قد يطول فعلياً لأجل غير مسمى
ج) Starvation أخطر من Deadlock دائماً
د) Deadlock يحدث فقط مع مسيّر واحد

**التعليل:** ب) ✅ هذا هو التمييز الدقيق المطلوب. أ) خاطئ، هناك فرق جوهري. ج) لا مقارنة مطلقة بالخطورة بهذا الشكل العام. د) خاطئ، Deadlock يحدث عادة مع أكثر من مورد/مسيّر (كما في مثال P0/P1).

## الجزء الرابع: أسئلة تصحيح الكود (خوارزميات)

### تصحيح 1 — نوع: logic
**الكود الخاطئ:**
```c
// Producer (buggy order)
wait(mutex);       // enter critical region first
wait(empty);       // then check for empty slot
insert_item();
signal(mutex);
signal(full);
```
**اكتشف الخطأ:** تم عكس ترتيب `wait(mutex)` و`wait(empty)`؛ فإن كان المخزن ممتلئاً بالكامل (`empty==0`)، ستُحجب العملية على `wait(empty)` بينما هي لا تزال **حاجزة `mutex`** بالفعل من `wait(mutex)` السابقة — مما يمنع أي مستهلك من الدخول لتحرير مكان عبر `signal(empty)`، فيحدث **استعصاء تام**.
**التصحيح:**
```c
wait(empty);       // check for empty slot FIRST
wait(mutex);       // THEN enter critical region
insert_item();
signal(mutex);
signal(full);
```
**شرح الحل (3 نقاط):**
1. يجب دائماً استدعاء المسيّر العدّاد (`empty`/`full`) قبل مسيّر الإقصاء التبادلي (`mutex`).
2. عكس الترتيب يعني أن العملية قد تُحجب وهي لا تزال حاجزة لـ `mutex`، مما يمنع أي عملية أخرى من العمل على تحرير الشرط الذي تنتظره.
3. هذا مثال دقيق يوضح كيف أن ترتيب استدعاء المسيّرات المتعددة لا يقل أهمية عن صحة كل مسيّر بمفرده.

### تصحيح 2 — نوع: race_condition
**الكود الخاطئ:**
```c
// Reader process (missing protection on readcount)
readcount++;                  // no mutex protection!
if (readcount == 1) wait(wrt);
// reading is performed
readcount--;                   // no mutex protection!
if (readcount == 0) signal(wrt);
```
**اكتشف الخطأ:** تم حذف `wait(mutex)`/`signal(mutex)` المحيطتين بتعديل `readcount`؛ فإذا قرأ قارئان قيمة `readcount` وعدّلاها في نفس اللحظة تقريباً (تبديل سياق بينهما)، فقد تُفقد إحدى الزيادتين أو النقصانين (حالة سباق كلاسيكية على متغير مشترك)، مما قد يؤدي لعدم استدعاء `wait(wrt)` إطلاقاً رغم أن قارئاً هو الأول فعلياً (فيتمكن كاتب من الدخول أثناء وجود قرّاء نشطين!) أو العكس.
**التصحيح:**
```c
wait(mutex);
readcount++;
if (readcount == 1) wait(wrt);
signal(mutex);
// reading is performed
wait(mutex);
readcount--;
if (readcount == 0) signal(wrt);
signal(mutex);
```
**شرح الحل (3 نقاط):**
1. `readcount` متغير مشترك بين عدة قرّاء، ويجب حمايته بمسيّر إقصاء تبادلي (`mutex`) عند أي تعديل عليه.
2. غياب هذه الحماية يفتح الباب لحالة سباق كلاسيكية تُفسد منطق "أول قارئ يحجز، آخر قارئ يحرر".
3. هذا الخطأ خطير بشكل خاص لأنه قد لا يظهر إلا نادراً (فقط عند تزامن دقيق)، مما يجعله صعب الاكتشاف أثناء الاختبار العادي.

### تصحيح 3 — نوع: deadlock_misuse
**الكود الخاطئ:**
```c
// Philosopher i (attempt to fix naive solution — still buggy)
wait(chopStick[i]);
wait(chopStick[i]);   // typo: locking the same chopstick twice
eat();
signal(chopStick[i]);
signal(chopStick[(i + 1) % 5]);
```
**اكتشف الخطأ:** استُدعي `wait(chopStick[i])` مرتين متتاليتين بدل حجز العصا اليسرى ثم اليمنى (`chopStick[(i+1)%5]`)؛ هذا يعني أن الفيلسوف يحاول حجز **نفس العصا مرتين**، وبما أنها مسيّر ثنائي مُهيَّأ إلى 1، فإن الاستدعاء الثاني لـ `wait` على نفس العصا (التي أصبحت قيمتها 0 بعد الأول) سيحجب الفيلسوف إلى الأبد بانتظار نفسه — استعصاء ذاتي فوري.
**التصحيح:**
```c
wait(chopStick[i]);              // left chopstick
wait(chopStick[(i + 1) % 5]);    // right chopstick (corrected)
eat();
signal(chopStick[i]);
signal(chopStick[(i + 1) % 5]);
```
**شرح الحل (3 نقاط):**
1. يجب التأكد من أن كل نداء `wait` يستهدف مورداً مختلفاً فعلياً وليس نفس المورد بالخطأ.
2. حجز نفس المسيّر مرتين متتاليتين (بدون `signal` بينهما) هو أحد الأخطاء الأربعة الشائعة الموثقة صراحة في المحاضرة.
3. في هذا السياق تحديداً (موارد متعددة في مصفوفة)، يجب التحقق الدقيق من فهرسة كل مورد (`i` مقابل `(i+1)%5`).

### تصحيح 4 — نوع: misconception
**الكود الخاطئ (تعليق طالب):** "بما أن Monitor يضمن الإقصاء التبادلي تلقائياً، فلا حاجة لاستخدام `wait()`/`signal()` على أي متحول شرط داخل المراقب أبداً؛ يكفي الاعتماد على الحماية التلقائية."
**اكتشف الخطأ:** هذا فهم خاطئ شائع؛ الإقصاء التبادلي (منع دخول أكثر من عملية واحدة للمراقب في نفس اللحظة) **لا يحل** مشكلة الحاجة للانتظار حتى تحقق شرط معين (كما في Producer-Consumer: ماذا لو كان المخزن ممتلئاً؟). المراقب وحده يضمن فقط عدم التداخل، لكنه لا يضمن "الانتظار الذكي" بدون استخدام `Condition Variables` صراحة.
**التصحيح:** يجب دائماً استخدام `condition variable` مع `wait()`/`signal()` عليها عندما تحتاج العملية للانتظار حتى تحقق شرط منطقي داخل المراقب (كما في أمثلة `BoundedBuffer` و`Dining Philosophers` بالمراقبات).
**شرح الحل (3 نقاط):**
1. الإقصاء التبادلي والانتظار الشرطي مشكلتان مختلفتان تماماً، ولكل منهما آلية منفصلة حتى داخل Monitor.
2. نسيان استخدام `Condition Variables` عند الحاجة الفعلية للانتظار يؤدي لمنطق برنامج خاطئ (كأن يحاول المنتج الإدراج في مخزن ممتلئ دون أي آلية انتظار).
3. تذكر دائماً: Monitor = إقصاء تبادلي تلقائي + أدوات انتظار شرطي (Condition Variables) يجب استخدامها يدوياً عند الحاجة.

### تصحيح 5 — نوع: off_by_one
**الكود الخاطئ:**
```java
public synchronized void produce(int item) throws InterruptedException {
    if (buffer.size() == capacity) {   // using if instead of while
        wait();
    }
    buffer.add(item);
    notifyAll();
}
```
**اكتشف الخطأ:** استُخدم `if` بدل `while` لفحص امتلاء المخزن؛ إن استيقظ هذا الخيط (بسبب `notifyAll()` من مستهلك آخر) بينما خيط منتج آخر تمكن من ملء المخزن مجدداً قبله (تسابق بين عدة منتجين مستيقظين معاً)، فسينفذ `buffer.add(item)` رغم أن المخزن قد يكون ممتلئاً فعلياً بحلول وقت استيقاظه — تجاوز لسعة المخزن (`buffer overflow` منطقي).
**التصحيح:**
```java
public synchronized void produce(int item) throws InterruptedException {
    while (buffer.size() == capacity) {   // re-check condition after waking up
        wait();
    }
    buffer.add(item);
    notifyAll();
}
```
**شرح الحل (3 نقاط):**
1. `notifyAll()` يوقظ **جميع** الخيوط المنتظرة، وليس بالضرورة الخيط "المستحق" فقط للاستمرار.
2. الفحص بـ `while` يضمن إعادة التحقق من الشرط فور الاستيقاظ، وليس الافتراض بأن الاستيقاظ يعني تلقائياً تحقق الشرط.
3. هذه القاعدة (استخدام `while` لا `if`) عامة وإلزامية في كل استخدامات `Condition Variables`، سواء في Java أو في أي لغة أخرى تدعم Monitors.

### تصحيح 6 — نوع: race_condition
**الكود الخاطئ:**
```c
// Barber process (missing mutex protection)
down(customers);
waiting = waiting - 1;    // no mutex protection around this line!
up(barbers);
cut_hair();
```
**اكتشف الخطأ:** تم حذف `down(mutex)`/`up(mutex)` المحيطتين بتعديل `waiting`؛ فإذا كان زبون يصل في نفس لحظة قيام الحلاق بإنقاص `waiting`، قد يحدث تداخل على هذا المتغير المشترك (حالة سباق)، فيصبح العداد غير دقيق (قد يسمح بدخول عدد زبائن أكثر من سعة الكراسي الفعلية، أو يرفض زبوناً كان هناك مكان له فعلياً).
**التصحيح:**
```c
down(customers);
down(mutex);              // protect access to 'waiting'
waiting = waiting - 1;
up(barbers);
up(mutex);
cut_hair();
```
**شرح الحل (3 نقاط):**

### case_study — تمرين 1: العنوان: استعصاء عشاء الفلاسفة الفعلي
**السيناريو:** 5 فلاسفة ينفذون الحل الساذج بالمسيّرات المنفصلة لكل عصا. في لحظة معينة، تمكن كل الفلاسفة الخمسة من تنفيذ `wait(chopStick[i])` (حجز العصا اليسرى) بنجاح تام في نفس الدورة الزمنية تقريباً.
**المطلوب:**
1. حدد الشروط الأربعة الكلاسيكية للاستعصاء وكيف تنطبق هنا (Mutual Exclusion، Hold and Wait، No Preemption، Circular Wait).
2. اقترح حلاً عملياً واحداً بديلاً عن حل المراقبات (تم شرحه سابقاً) يمنع هذا السيناريو.

**نموذج الحل:**
1. `Mutual Exclusion`: كل عصا (مسيّر ثنائي) لا يمكن أن يحجزها إلا فيلسوف واحد في كل مرة. `Hold and Wait`: كل فيلسوف يحمل عصاه اليسرى وينتظر اليمنى دون تحرير ما يحمله. `No Preemption`: لا يمكن انتزاع العصا من فيلسوف قسراً. `Circular Wait`: الفيلسوف 0 ينتظر عصا الفيلسوف 1، والفيلسوف 1 ينتظر عصا الفيلسوف 2، وهكذا دورانياً حتى يعود الانتظار للفيلسوف 0 — سلسلة دائرية مغلقة.
2. حل بديل (كسر التناظر): جعل فيلسوف واحد فقط (مثلاً الفيلسوف رقم 0) يحجز عصاه **اليمنى أولاً ثم اليسرى**، بعكس بقية الفلاسفة الذين يحجزون اليسرى أولاً؛ هذا يكسر إمكانية نشوء الانتظار الدائري لأن أحد الفلاسفة سيحاول حجز عصا مختلفة الترتيب عن جيرانه.

## الجزء الرابع: أسئلة تصميم (هيكلية)

### سؤال تصميم 1: تصميم حل Readers-Writers بأفضلية للكاتب (Writer-Priority) بدل القرّاء
**المطلوب:** الحل المذكور في المحاضرة يُعطي أفضلية للقرّاء وقد يُجوّع الكاتب. صمم تعديلاً يمنع القرّاء الجدد من الدخول إن كان هناك كاتب ينتظر بالفعل، لضمان عدالة أكبر تجاه الكاتب.

**نموذج الإجابة:**
```c
Semaphore mutex = 1;       // protects readcount
Semaphore wrt = 1;         // writer's lock
Semaphore queue = 1;       // NEW: ensures fairness — writer waiting blocks new readers
int readcount = 0;

// Reader
wait(queue);        // wait in the same fairness queue as writers
wait(mutex);
readcount++;
if (readcount == 1) wait(wrt);
signal(mutex);
signal(queue);
// reading is performed
wait(mutex);
readcount--;
if (readcount == 0) signal(wrt);
signal(mutex);

// Writer
wait(queue);        // writer also waits in the same fairness queue
wait(wrt);
signal(queue);
// writing is performed
signal(wrt);
```

**معايير التقييم:**
- إضافة مسيّر `queue` مشترك يمر عبره كل من القرّاء والكاتب بنفس الترتيب الزمني للوصول (FIFO تقريبي).
- ضمان أن قارئاً جديداً لا يمكنه "القفز" أمام كاتب ينتظر بالفعل في `queue`.
- الحفاظ على منطق `readcount`/`wrt` الأصلي دون تغيير جوهري في آلية القراءة المتوازية.

### سؤال تصميم 2: تصميم حل Sleeping Barber بأكثر من حلاق واحد (Multiple Barbers)
**المطلوب:** الحل المذكور في المحاضرة يفترض حلاقاً واحداً فقط (`single sleeping barber problem`). صمم تعديلاً يسمح بوجود عدد `K` من الحلاقين يعملون بالتوازي على خدمة نفس رتل الزبائن المشترك.

**نموذج الإجابة:**
```c
#define CHAIRS 5
#define NUM_BARBERS 3        // K barbers
semaphore customers = 0;
semaphore barbers = 0;
semaphore mutex = 1;
int waiting = 0;

void Barber(void) {           // each of the K barber threads runs this same code
    while (TRUE) {
        down(customers);       // any free barber may wake up here
        down(mutex);
        waiting = waiting - 1;
        up(barbers);
        up(mutex);
        cut_hair();
    }
}

void customer(void) {
    down(mutex);
    if (waiting < CHAIRS) {
        waiting = waiting + 1;
        up(customers);          // wakes up ANY one of the K available barbers
        up(mutex);
        down(barbers);
        get_haircut();
    } else {
        up(mutex);
    }
}
```

**معايير التقييم:**
- تشغيل `K` نسخة من دالة `Barber` نفسها كخيوط/عمليات مستقلة تتنافس جميعها على نفس مسيّرات `customers`/`barbers`/`mutex` المشتركة.
- عدم الحاجة لأي تعديل جوهري على منطق المسيّرات نفسه، لأن `Counting Semaphore` يدعم بطبيعته تعدد "المستهلكين" (الحلاقين هنا) للإشارات دون تعديل إضافي.
- التأكيد على أن `mutex` يبقى مسيّراً ثنائياً واحداً يحمي `waiting` بصرف النظر عن عدد الحلاقين، لأن التعديل على هذا المتغير المشترك يبقى عملية واحدة حرجة بغض النظر عن عدد من يقرأها/يعدلها.

## الجزء الخامس: أسئلة نظرية متوقعة بالامتحان

### السؤال 1: عدد المسيّرات المطلوبة لحل Producer-Consumer الكامل، ودور كل واحد منها.
**نموذج الإجابة:** ثلاثة مسيّرات: `empty` (Counting = N) لعدّ الأماكن الفارغة، `full` (Counting = 0) لعدّ العناصر الجاهزة، `mutex` (Binary = 1) لحماية عملية الإدراج/السحب من التداخل بين عمليات متعددة من نفس النوع.

### السؤال 2: اشرح الفكرة الذكية وراء استخدام `readcount` مع `wrt` في حل Readers-Writers.
**نموذج الإجابة:** بدل أن يحجز كل قارئ `wrt` بمفرده (مما يمنع التوازي)، يحجزه القارئ الأول فقط (`readcount==1`) نيابة عن كل مجموعة القرّاء النشطين، ويحرره القارئ الأخير فقط (`readcount==0` بعد الإنقاص)، فيعمل `wrt` كقفل بين "مجموعة القرّاء" و"الكاتب" لا بين كل قارئ فردي والكاتب.

### السؤال 3: ما تعريف Deadlock الرسمي المذكور في المحاضرة؟
**نموذج الإجابة:** حالة تكون فيها عمليتان أو أكثر جميعها في حالة انتظار لحدث ما، ولا يمكن لهذا الحدث أن يتحقق إلا من قبل عملية أخرى من نفس مجموعة العمليات المنتظرة.

### السؤال 4: ما الفرق بين Deadlock وStarvation؟
**نموذج الإجابة:** Deadlock انتظار دائري لا ينفك أبداً بدون تدخل خارجي (كل الأطراف منتظرة بشكل متبادل)، بينما Starvation دخول عملية في انتظار طويل قد ينتهي نظرياً (العملية ليست منتظرة لعملية أخرى منتظرة بالضرورة) لكنه قد يطول لأجل غير مسمى بسبب سياسة اختيار غير عادلة.

### السؤال 5: عرّف Priority Inversion واشرح كيف يحل Priority Ceiling هذه المشكلة.
**نموذج الإجابة:** Priority Inversion هي مشكلة جدولة تجعل عملية عالية الأولوية تنتظر بسبب عملية منخفضة الأولوية تحمل مورداً تحتاجه (وقد تتفاقم بتدخل عملية متوسطة الأولوية تسحب المعالج من صاحب المورد منخفض الأولوية). يحلها Priority Ceiling برفع أولوية حامل القفل مؤقتاً لتكون أعلى من أي طالب محتمل له، فيمنع أي عملية متوسطة من مقاطعته.

### السؤال 6: لماذا يحدث استعصاء محتمل في الحل الساذج لمسألة عشاء الفلاسفة بالمسيّرات المنفصلة؟
**نموذج الإجابة:** إن حجز كل الفلاسفة عصاهم اليسرى في نفس اللحظة تقريباً، فستكون كل عصا يمنى محجوزة من الفيلسوف المجاور، فينتظر الجميع بشكل دائري إلى الأبد — انتظار دائري كلاسيكي على موارد متعددة.

### السؤال 7: كيف يتفادى حل عشاء الفلاسفة بالمراقبات هذا الاستعصاء؟
**نموذج الإجابة:** لا يحجز أي فيلسوف أي عصا بشكل منفصل؛ بدلاً من ذلك، يفحص المراقب حالة الجارين (`test`) قبل السماح لأي فيلسوف بالانتقال لحالة `EATING`، فإما يأكل الفيلسوف بكلا العصاتين منطقياً معاً أو لا يحجز شيئاً إطلاقاً وينتظر — فلا يوجد سيناريو "حجز نصفي" يؤدي لانتظار دائري.

### السؤال 8: اذكر الأخطاء الأربعة الشائعة في استخدام المسيّرات وأثر كل منها.
**نموذج الإجابة:** 1) عكس الترتيب (`signal` قبل `wait`) → فشل الإقصاء التبادلي. 2) تكرار `wait` مرتين متتاليتين → استعصاء. 3) حذف `wait` → فشل الإقصاء التبادلي. 4) حذف `signal` → استعصاء.

### السؤال 9: ما تعريف Monitor، وما الميزة الجوهرية التي يوفرها؟
**نموذج الإجابة:** Monitor هو مجموعة إجراءات ومتحولات وبنى معطيات مجمّعة في وحدة نمطية خاصة، بحيث لا يمكن الوصول للبيانات الداخلية إلا عبر إجراءاته. الميزة الجوهرية: ضمان الإقصاء التبادلي تلقائياً (عملية واحدة فقط نشطة داخل المراقب في أي لحظة)، يتولاه المترجم لا المبرمج.

### السؤال 10: لماذا نحتاج Condition Variables رغم أن Monitor يضمن الإقصاء التبادلي؟
**نموذج الإجابة:** لأن الإقصاء التبادلي وحده لا يكفي لحل مسائل تحتاج "انتظاراً حتى يتحقق شرط" (كمخزن ممتلئ أو فارغ في Producer-Consumer)؛ لذلك أُضيفت Condition Variables مع `wait()`/`signal()` عليها للسماح بإيقاف عملية مؤقتاً والسماح لعملية أخرى بدخول المراقب في نفس الوقت.

### السؤال 11: ما الفرق بين متحول الشرط والمسيّر من ناحية تذكر الإشارات؟
**نموذج الإجابة:** المسيّر يحتفظ بقيمة عددية تراكمية، فيمكن أن تتجاوز الصفر حتى بدون منتظرين فعليين، وتُستهلك لاحقاً؛ أما متحول الشرط فلا يملك عداداً إطلاقاً، فإن استُدعي `signal()` بدون وجود عملية منتظرة فعلياً في تلك اللحظة، تضيع الإشارة تماماً بلا أي أثر.

### السؤال 12: لماذا يجب استخدام `while` بدل `if` عند الانتظار على شرط داخل Monitor؟
**نموذج الإجابة:** لأن الخيط المستيقظ (خاصة عبر `notifyAll()` الذي يوقظ الجميع) قد لا يجد شرطه متحققاً فعلياً وقت استيقاظه بسبب تسابق خيوط أخرى استهلكت الفرصة قبله؛ فاستخدام `while` يضمن إعادة فحص الشرط دائماً بعد كل استيقاظ قبل المتابعة.

## الجزء الرابع: بطاقات سؤال وجواب (Q&A Cards)

**Q1:** ما الاسم البديل لمسألة Producer-Consumer؟
A: مسألة المخزن المحدود (`Bounded-Buffer Problem`).

**Q2:** ما القيم الابتدائية لمسيّرات empty وfull وmutex في حل Producer-Consumer؟
A: empty=N، full=0، mutex=1.

**Q3:** من يحجز مسيّر wrt في حل Readers-Writers؟
A: القارئ الأول فقط (عندما readcount يصبح 1)، والكاتب أيضاً عند رغبته بالكتابة.

**Q4:** أي عملية معرّضة للتجويع في حل Readers-Writers المذكور؟
A: الكاتب، بسبب استمرار وصول قرّاء جدد بلا انقطاع.

**Q5:** ما تعريف Deadlock؟
A: حالة تكون فيها كل عملية من مجموعة عمليات في انتظار حدث لا يمكن أن يتحقق إلا من قبل عملية أخرى من نفس المجموعة المنتظرة.

**Q6:** ما تعريف Starvation؟
A: الدخول في حالة انتظار لا نهائي دون أن تُنقل العملية أبداً لحالة الجاهزية.

**Q7:** ما تعريف Priority Inversion؟
A: مشكلة جدولة تجعل عملية عالية الأولوية تنتظر بسبب عملية منخفضة الأولوية تحمل قفلاً أو مسيّراً تحتاجه.

**Q8:** كيف يحل Priority Ceiling مشكلة عكس الأولوية؟
A: برفع أولوية حامل القفل مؤقتاً لتكون أعلى من أي طالب محتمل له.

**Q9:** لماذا يستعصي الحل الساذج لعشاء الفلاسفة بالمسيّرات المنفصلة؟
A: لأن كل فيلسوف قد يحجز عصاه اليسرى أولاً، فتُحجز كل العصي اليمنى من الجيران، وينتظر الجميع بشكل دائري.

**Q10:** كيف يتفادى حل المراقبات هذا الاستعصاء؟
A: بفحص حالة الجارين قبل السماح بالأكل؛ الفيلسوف إما يأكل بكلا العصاتين منطقياً أو لا يحجز شيئاً وينتظر.

**Q11:** كم مسيّراً يُستخدم في حل الحلاق النائم، وما أسماؤها؟
A: ثلاثة: customers، barbers، mutex (accessSeats).

**Q12:** هل حل الحلاق النائم المذكور معرّض للاستعصاء أم للتجويع؟
A: للتجويع (Starvation)، وليس للاستعصاء.

**Q13:** ما تعريف Monitor؟
A: مجموعة إجراءات ومتحولات وبنى معطيات مجمّعة في وحدة نمطية خاصة، لا يمكن الوصول لبياناتها الداخلية إلا عبر إجراءاتها.

**Q14:** ما الميزة الجوهرية للمراقب؟
A: ضمان الإقصاء التبادلي تلقائياً؛ عملية واحدة فقط نشطة داخل المراقب في أي لحظة، يديرها المترجم لا المبرمج.

**Q15:** لماذا نحتاج Condition Variables مع المراقبات؟
A: لأن الإقصاء التبادلي وحده لا يكفي لحل مسائل تحتاج انتظاراً حتى يتحقق شرط معين.

**Q16:** ما الفرق بين متحول الشرط والمسيّر من ناحية تذكر الإشارات؟
A: المسيّر يحتفظ بقيمة عددية تراكمية، بينما متحول الشرط لا يملك عداداً فتضيع الإشارة إن لم يوجد منتظر فعلي.

**Q17:** لماذا نستخدم while بدل if عند الانتظار على شرط داخل Monitor؟
A: لأن الخيط المستيقظ قد لا يجد شرطه متحققاً فعلياً بعد الاستيقاظ بسبب تسابق خيوط أخرى، فيجب إعادة الفحص.

**Q18:** ما أثر عكس ترتيب signal ثم wait على mutex؟
A: فشل الإقصاء التبادلي تماماً — تتمكن كل العمليات من الدخول إلى مناطقها الحرجة بحرية.
